## 🏗 Архітектурні рішення та відхилення від ТЗ

Під час виконання завдання було виявлено декілька логічних суперечностей у початкових вимогах. Для забезпечення безпеки та відповідності сучасним стандартам розробки (RESTful, SOLID), було прийнято наступні рішення:

1. **RESTful Routing:**
    * *Завдання:* Єдиний ендпоінт `/v1/api/users` для всіх методів, де `id` передається у вхідних атрибутах.
    * *Рішення:* У реалізації використано REST-підхід:
         `GET /v1/api/users/{id}`
         `PUT /v1/api/users/{id}`
         `DELETE /v1/api/users/{id}`
    * Причина: для REST API ідентифікатор ресурсу природніше передавати в URL, а не в request body.
2. **Унікальний індекс тільки на `login`:**
    * *Завдання:* Складовий унікальний індекс на `login` та `pass`.
    * *Рішення:* Складовий індекс дозволив би реєстрацію різних користувачів з однаковим логіном, але різними паролями, що ламає механізм аутентифікації. Унікальний індекс встановлено виключно на поле `login`.
    * Фактична логіка:
      -один login = один користувач
    Для цього використовується:
      validation-level перевірка через UniqueEntity;
      database-level unique constraint для login.
3. **Хешування, зберігання та довжина паролів:**
    * *Завдання:* Зберігати пароль та обмежити його довжину до 8 символів у базі.
    * *Рішення:* У ТЗ поле називається pass. В API це поле також називається pass, щоб відповідати формату завдання.
         Але всередині застосунку використовується назва password, а в базі даних зберігається не сирий пароль, а його hash. 
         Довжина до 8 символів перевіряється через Symfony Validator на етапі DTO. У самій базі даних поле має довжину 255 символів для збереження надійного хешу.
         Тобто при створенні або оновленні користувача:
         pass з request body → hash → password у базі
    * У ТЗ в response для POST та GET вказане поле pass.
         Щоб не відходити від формату відповіді, поле pass повертається.
         Але важливо: це не оригінальний пароль користувача, а захешоване значення з бази даних.
         Це зроблено тільки для відповідності формату тестового завдання. У реальному production API пароль або його hash не варто повертати у response.
4. **Логіка створення (POST) та ролі:**
    * *Завдання:* Роль `user` має право робити POST "в рамках свого користувача".
    * *Рішення:* Оскільки API повністю закрите зовні (потребує Bearer-токен), неавторизований користувач не може зробити запит на своє створення. Тому право на створення користувачів (`POST`) делеговано виключно ролі `root`. Роль `user` може лише переглядати (`GET`) та оновлювати (`PUT`) свій власний профіль.
5. **Централізована обробка помилок:**
    * Усі винятки (помилки валідації DTO, `UniqueConstraintViolationException` від БД при спробі створити дублікат, помилки доступу) перехоплюються глобальним `ApiExceptionSubscriber` та форматуються у стандартизований JSON без стектрейсу, як і вимагалося в ТЗ.


## Проєкт можна повністю запустити через Docker.

## Авторизація

API використовує Bearer JWT.

Приклад заголовка:

Authorization: Bearer <token>

Для отримання токена використовується endpoint:

POST `/v1/api/login`

Body:

`{
"login": "root",
"pass": "rootpass"
}`

У відповіді повертається JWT-токен.

# Не забудьте згенерувати JWT ключі: bin/console lexik:jwt:generate-keypair

## Тестування
** Оскільки створення користувачів через API доступне тільки користувачу з правами ROLE_ROOT, для первинного створення root-користувача передбачена консольна команда: `bin/console app:create-root`
** Для ручного тестування в проєкті є файл: `requests.http`
# У файлі є готові запити для перевірки:
- Створення звичайного користувача від імені root.
- Авторизація під звичайним користувачем.
- Читання та оновлення свого профілю звичайним користувачем.
- Перевірка заборон: спроба видалити себе, спроба створити або оновити іншого користувача.
- Успішне видалення звичайного користувача від імені root.
- Перевірка запитів без JWT-токена.
- Перевірка валідації (помилки 422 Unprocessable Entity у форматі JSON).
